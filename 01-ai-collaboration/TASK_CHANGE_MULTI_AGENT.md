# 任务、变更与多 Agent 协议

## 1. 标准任务契约

任何实施任务必须包含：

- 任务 ID 和关联需求 ID
- 目标用户结果
- 当前行为和期望行为
- 允许修改的领域、模块和文件范围
- 明确禁止修改的内容
- 输入、输出、依赖和外部服务
- 数据迁移或兼容性影响
- 验收标准和测试列表
- 回滚方式
- 变更预算

## 2. 变更预算

默认限制：

- 一个任务只解决一个主要目标
- 一个 PR 对应一个可解释变更
- 修改跨越多个领域时必须拆分或升级为架构任务
- 无关格式化、重命名和重构不与功能修改混合
- AI 在发现额外问题时创建风险或后续任务，不顺手修改

实际文件数和代码行阈值应由项目根据规模确定；原则是审查者能够完整理解和回滚。

## 3. 实施前影响分析

AI 必须列出：

- 直接调用者和被调用者
- 数据读写和迁移
- 公共接口和兼容性
- 权限和租户边界
- 缓存、队列和后台任务
- 日志、指标和告警
- 需要新增或更新的测试

## 4. 问题、计划和修改严格分离

- `ANSWER`：只回答，不编辑。
- `DESIGN`：允许输出方案和补丁计划，不写入代码。
- `IMPLEMENT`：只执行用户批准的任务契约。
- `VERIFY`：不得为了让测试通过而自动改变需求或核心业务语义。

## 5. 多 Agent 使用条件

默认单 Agent。只有同时满足以下条件才并行：

- 任务能够按领域或模块独立
- 每个 Agent 有不同所有权范围
- 共享接口已冻结或版本化
- 每个分支可以独立测试
- 有明确集成人负责最终合并

## 6. 多 Agent 执行规则

1. 每个 Agent 使用独立分支和 Git Worktree。
2. 每个任务声明模块所有权和预计文件 footprint。
3. 禁止两个 Agent 同时修改同一业务不变量或数据库迁移链。
4. 公共接口先由架构任务批准，再分配实现。
5. 分支按依赖顺序逐个合并，每次合并后运行完整门禁。
6. 集成人只能解决合并和接口问题，不能无批准重写功能。
7. 冲突意味着任务拆分或设计有问题，不应全部交给 AI 自动猜测。

## 7. 失败恢复

- 保留任务前基线 Commit。
- 每个可验证步骤形成小 Commit。
- 失败时回退到最后一个通过门禁的状态。
- 不在损坏工作区中连续提示模型“再试一次”。
