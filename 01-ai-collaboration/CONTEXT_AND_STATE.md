# 上下文与项目状态协议

## 1. 三层上下文

### 稳定上下文

长期变化较少：项目目标、领域模型、架构边界、安全不变量、正式决定。

来源：`PROJECT.md`、`PRD.md`、`ARCHITECTURE.md`、决定记录。

### 当前状态

持续变化：当前版本、里程碑、已完成任务、阻塞、已知缺陷和生产状态。

来源：`PLAN_AND_STATE.md`。

### 任务上下文

只服务本次任务：关联需求、涉及模块、代码入口、接口、测试、日志和限制。

每次任务开始时临时组装，任务结束后丢弃或压缩成证据和状态。

## 2. 不要注入整个项目

AI 应先建立检索路径：

```text
需求 ID → 领域 → 模块 → 接口/数据 → 文件 → 测试
```

只有命中当前任务的内容才进入上下文。无差别提供整个仓库会增加噪声、旧信息和错误关联。

## 3. 根级 AI 规则保持最小

根规则只保留：

- 项目文档入口
- 构建和测试命令
- 最高优先级不变量
- 禁止修改的区域
- 安全和密钥要求

具体业务规则放在 PRD，模块规则放在架构或模块文档。不要自动生成大段通用规范填充根规则。

## 4. 状态快照

每次任务结束必须更新：

- 当前版本与分支
- 任务状态
- 修改和未修改内容
- 运行过的测试
- 证据链接
- 新风险和决定
- 下一项唯一任务

状态只记录真实发生的事情，不复制聊天总结中的推测。

## 5. 新会话交接

新会话开始时，先让 AI 回答：

1. 项目目标是什么
2. 当前阶段和版本是什么
3. 当前任务是什么
4. 受影响的领域和模块是什么
5. 哪些决定不能推翻
6. 哪些证据仍然缺失

用户确认摘要正确后再实施。

## 6. 上下文预算

- 优先接口、类型、数据模型和不变量，而不是大段实现。
- 优先最近通过验证的文档和代码。
- 超过一个领域的任务先拆分。
- 上下文出现冲突时停止，不通过增加更多文本掩盖冲突。

研究依据：[C1]、[C2]、[C4]、[C10]、[R3]，见 [`../00-research/REFERENCES.md`](../00-research/REFERENCES.md)。
