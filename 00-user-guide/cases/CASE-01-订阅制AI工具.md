# 案例：我怎样用 AI 开发一个订阅制 AI 工具

> 面向：用户

这个案例展示我怎样把“做一个 AI 工具”的模糊想法，变成能够面对真实用户的项目。

## 1. 我的原始想法

> 我想做一个帮助内容创作者分析频道数据、发现选题机会并生成内容计划的网站，通过月度订阅收费。

如果我马上让 AI 写代码，它很可能先生成首页、登录和聊天框，但不会自动解决数据来源、权限、成本、支付、限额和用户留存。

## 2. 我先确认项目价值

AI 通过采访帮我得到：

| 问题 | 当前答案 |
|---|---|
| 目标用户 | 有一定内容产量，但不会系统分析数据的中小创作者 |
| 真实问题 | 数据分散，选题依靠感觉，复盘耗时 |
| 核心价值 | 把频道数据转换成可执行选题和内容计划 |
| 付费理由 | 节省分析时间，提高内容命中率 |
| V1 闭环 | 连接频道 → 同步数据 → 生成分析 → 保存选题 → 订阅继续使用 |
| 暂时不做 | 团队协作、自动发布、多平台全覆盖 |

这些内容写入 `PROJECT.md`。

## 3. 我定义产品，而不是只列页面

AI 帮我定义角色和流程：

```mermaid
flowchart LR
    A[注册] --> B[连接频道]
    B --> C[同步公开视频数据]
    C --> D[生成分析]
    D --> E[保存选题]
    E --> F[达到免费额度]
    F --> G[订阅]
    G --> H[继续分析]
```

同时写清失败情况：

- 频道授权失败；
- 数据源限流；
- 分析任务超时；
- 用户重复提交；
- 订阅支付失败；
- 用户取消订阅；
- 第三方 API 暂时不可用。

这些内容写入 `PRD.md`。

## 4. 我让 AI 建立领域和架构

核心领域可能包括：

- Identity：用户与会话；
- Channel：频道连接和同步；
- Analytics：数据分析任务；
- Ideas：选题与内容计划；
- Billing：套餐、额度、订阅和付款状态；
- Admin：客服、审计和异常处理。

架构先采用模块化单体，而不是一开始拆成很多微服务。

AI 还需要设计：

- 异步分析任务；
- 第三方 API 限流和重试；
- 用户数据隔离；
- 分析结果缓存；
- 每次分析的 AI 成本；
- 日志、指标和错误追踪。

这些内容写入 `ARCHITECTURE.md`。

## 5. 我把开发拆成可验证切片

我不会让 AI 一次开发全部系统，而是按顺序推进：

1. 用户注册与登录；
2. 连接一个频道；
3. 同步最小数据集；
4. 创建一次分析任务；
5. 展示并保存分析结果；
6. 记录免费额度；
7. 创建订阅；
8. 处理取消和支付失败；
9. 增加后台和监控；
10. 完成发布准备。

每个任务都写入 `PLAN_AND_STATE.md`，并声明允许修改的模块、验收标准、测试和回退方式。

## 6. 我怎样判断一个任务完成

以“创建分析任务”为例：

- 代码生成：`GENERATED`；
- 本地任务执行成功：`EXECUTED`；
- 正常、超时、重复提交和额度不足测试通过：`VERIFIED`；
- 部署到生产环境：`RELEASED`；
- 真实用户使用后任务成功率和成本稳定：`OBSERVED`。

证据写入 `DECISIONS_RISKS_EVIDENCE.md`。

## 7. 我上线前重点检查

- 不同用户不能读取彼此频道数据；
- 第三方 Token 不出现在仓库和日志；
- 同一分析请求不会重复扣额度；
- 支付回调经过验证并且可以安全重试；
- AI 和数据 API 成本有配额与告警；
- 分析失败时用户能看到状态并重试；
- 数据库有备份，迁移可以回退；
- 生产日志能关联用户、任务和请求。

## 8. 我从真实市场继续迭代

上线后我关注：

- 连接频道成功率；
- 首次分析完成率；
- 用户保存选题的比例；
- 免费转付费；
- 7 日和 30 日留存；
- 单次分析成本；
- 任务失败率；
- 用户最常反馈的问题。

下一版本由这些数据决定，而不是因为 AI 又想到了更多功能。

## 这个案例真正想说明什么

复杂项目不是靠一个巨大提示词完成的。

我需要和 AI 一起维护一条连续链路：

```text
市场价值
→ 产品需求
→ 领域与架构
→ 小型任务
→ 真实验证
→ 可靠发布
→ 用户数据
```

只要这条链路没有断，换模型、换聊天或换 AI 平台，都不会让项目重新从零开始。
