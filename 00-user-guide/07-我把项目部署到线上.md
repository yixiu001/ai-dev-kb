# 第 7 步：我把项目部署到线上

> 面向：第一次正式发布项目的用户

## 这一步完成后，我会得到什么

- 一个明确的部署目标；
- 测试、预发布和生产环境说明；
- 生产配置和环境变量清单；
- 数据备份和迁移计划；
- 发布步骤和回退步骤；
- 一个可以访问的线上地址；
- 上线后的真实验证结果。

## 我先确认一件事

“部署命令成功”不等于“产品已经上线”。

真正上线至少包括：

```mermaid
flowchart LR
    A[构建成功] --> B[部署成功]
    B --> C[环境配置正确]
    C --> D[数据库和外部服务正常]
    D --> E[线上核心流程通过]
    E --> F[日志与监控稳定]
```

## 第 1 个操作：选择部署平台

我让 AI 根据当前架构给出最多 3 个方案：

```text
现在进入 RELEASE 规划模式，但不要执行生产发布。

请读取 ARCHITECTURE.md 和当前代码结构。
根据以下条件给出最多 3 个部署方案：
- 我的预算；
- 当前技术栈；
- 数据库和文件存储；
- 是否需要后台任务；
- 是否需要自定义域名；
- 预计用户量；
- 我能承担的维护复杂度。

每个方案说明：
1. 费用；
2. 配置难度；
3. 支持的能力；
4. 主要限制；
5. 迁移和退出难度；
6. 推荐理由。
```

我选择的不一定是最强平台，而是最适合当前 V1 的平台。

## 第 2 个操作：填写 RELEASE.md

AI 应该帮助我完成：

- 目标版本和 Commit；
- 本地、测试、预发布和生产环境；
- 构建和启动命令；
- 数据库、存储、队列和外部服务；
- 环境变量名称；
- 域名和网络配置；
- 数据迁移；
- 日志、指标和告警；
- 回退条件和命令；
- 线上冒烟测试。

环境变量只记录名称和用途，不把真实密钥写入仓库。

## 第 3 个操作：先发布到预发布环境

预发布环境应该尽量接近生产：

- 相同运行时版本；
- 相同构建方式；
- 相同数据库类型；
- 相同外部服务模式；
- 不同的测试密钥和测试数据；
- 接近真实的数据规模。

我先在预发布环境完成：

- 构建；
- 数据迁移；
- 登录；
- 核心用户流程；
- 权限和数据隔离；
- 外部 API；
- 日志和告警；
- 回退演练。

## 第 4 个操作：执行上线前检查

- [ ] 目标 Commit 已固定；
- [ ] 构建和全部必要测试通过；
- [ ] 生产环境变量已配置；
- [ ] 数据库已备份；
- [ ] 迁移已在预发布验证；
- [ ] 权限和租户隔离测试通过；
- [ ] Secret 和依赖扫描通过；
- [ ] 日志、错误追踪和告警可用；
- [ ] 回退版本和回退步骤明确；
- [ ] 已知风险已经批准。

只要关键项未通过，我就不发布。

## 第 5 个操作：让 AI 输出发布 Runbook

```text
请根据 RELEASE.md 生成本次发布 Runbook。
每一步必须包含：
- 执行人或工具；
- 命令或操作；
- 预期结果；
- 证据；
- 失败时停止条件；
- 回退操作。

不要直接执行生产修改，先让我审查并批准 Runbook。
```

示例：

| 顺序 | 操作 | 预期结果 | 失败处理 |
|---|---|---|---|
| 1 | 备份数据库 | 生成可恢复备份 | 停止发布 |
| 2 | 构建固定 Commit | 构建完成且扫描通过 | 修复后重建 |
| 3 | 部署应用 | 健康检查通过 | 回退旧版本 |
| 4 | 执行迁移 | 验证查询正确 | 执行迁移回退 |
| 5 | 线上冒烟测试 | 核心流程通过 | 关闭流量并回退 |

## 第 6 个操作：批准并执行发布

只有在平台具备真实部署权限、我确认目标环境和风险后，才进入写入操作。

我可以说：

```text
我批准发布版本 <版本号 / Commit> 到 <环境>。
严格按已批准 Runbook 执行。
任何步骤与预期不一致时立即停止，不要自动跳过或继续。
执行后逐项返回真实工具结果。
```

## 第 7 个操作：线上冒烟测试

我至少验证：

- 线上地址和 HTTPS；
- 首页和静态资源；
- 注册、登录和会话；
- 核心用户旅程；
- 数据读写；
- 不同用户的数据隔离；
- 外部 API、通知和账单；
- 错误日志和告警；
- 回退入口。

## 第 8 个操作：观察生产状态

发布后我不会立即宣布结束。我设定观察窗口，例如 30 分钟、24 小时或一周，查看：

- 错误率；
- P95 响应时间；
- 核心任务成功率；
- 队列积压；
- 第三方服务错误；
- 用户反馈；
- 成本异常。

## AI 应该输出什么

- 实际发布版本；
- 实际执行步骤；
- 每一步结果；
- 线上地址；
- 冒烟测试结果；
- 日志和监控状态；
- 失败或跳过项；
- 当前最高可信状态；
- 回退是否仍可执行。

## 状态怎样判断

| 结果 | 状态 |
|---|---|
| 只生成部署说明 | `GENERATED` |
| 在测试环境运行 | `EXECUTED` |
| 预发布验收通过 | `VERIFIED` |
| 生产部署和冒烟测试通过 | `RELEASED` |
| 经过真实流量观察 | `OBSERVED` |

## 卡住时怎么办

### 部署平台显示成功，但网站打不开

检查域名、端口、启动命令、环境变量、健康检查和平台日志，不要重复盲目部署。

### 数据库迁移失败

停止继续发布，保护现有数据，使用预先准备的回退方案，不让 AI 临时猜测生产数据修复命令。

### 线上功能和本地不同

比较运行时版本、环境变量、数据库、外部回调地址、构建产物和网络权限。

## 完成检查

- [ ] 线上地址真实可访问；
- [ ] 生产版本和 Commit 已记录；
- [ ] 核心流程通过；
- [ ] 权限和数据隔离通过；
- [ ] 日志和告警可用；
- [ ] 回退方式可执行；
- [ ] 生产观察结果已记录；
- [ ] 状态没有被夸大。
