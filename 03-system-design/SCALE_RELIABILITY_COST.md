# 规模、可靠性与成本标准

## 1. 先写非功能目标

每个市场化项目至少定义：

- 预计注册用户、日活和峰值并发
- 关键接口延迟目标
- 可用性目标和允许停机时间
- 数据丢失容忍度 RPO
- 恢复时间目标 RTO
- 单用户、单请求或单任务成本上限
- 数据保留与合规要求

没有数字时写清假设和验证日期。

## 2. 渐进式扩展

### 初期

- 模块化单体
- 托管数据库和对象存储
- 明确索引和分页
- 基础限流、日志和错误追踪

### 用户增长后

根据真实指标引入：

- 缓存
- 异步队列和后台任务
- 读写隔离或分区
- 搜索服务
- 独立扩展的服务
- CDN 和边缘能力

不得因为“以后可能有大量用户”过早引入全部复杂基础设施。

## 3. 可靠性机制

- 所有外部调用设置超时
- 只对可安全重试的操作重试
- 写操作使用幂等键
- 队列任务记录状态、次数和死信
- 关键依赖具备降级体验
- 数据迁移可向前兼容并可回滚
- 定时备份并周期性恢复演练

## 4. 性能与容量验证

- 用真实数据规模测试查询和索引
- 对关键 API 进行负载测试
- 记录 P50、P95、P99 延迟和错误率
- 检查 N+1 查询、无限列表和大文件
- 验证高峰下的限流和队列堆积

## 5. 可观测性

每个关键用户旅程应能回答：

- 请求是否成功
- 失败在哪个模块或外部服务
- 哪个用户、租户、版本和请求 ID 受影响
- 错误何时开始、影响多大
- 是否需要回滚

至少具备结构化日志、错误追踪、核心指标和告警。

## 6. 成本控制

- 为 AI、邮件、存储、搜索和第三方 API 设置配额
- 记录每项能力的单位成本
- 防止无限重试、重复任务和恶意消耗
- 在发布前估算基准、10 倍和峰值用户成本
- 成本异常进入告警和事故流程
