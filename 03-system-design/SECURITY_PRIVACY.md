# 安全与隐私标准

功能可用不代表安全可用。安全必须独立设计和验证。

## 1. 威胁建模

对每个关键流程识别：

- 需要保护的数据和资产
- 攻击者与滥用者
- 信任边界
- 未授权读取、修改、删除和资源消耗路径
- 防护、检测和恢复方式

## 2. 身份认证与授权分离

- Authentication 只证明“是谁”。
- Authorization 决定“能访问谁的什么数据”。
- 每个数据库查询和 API 操作都必须有所有权或租户边界。
- 前端隐藏按钮不构成权限控制。
- 使用 RLS 时必须用两个真实用户进行交叉隔离测试。

## 3. 安全基线

- 密钥只放 Secret Manager 或环境变量
- 日志、错误和分析数据脱敏
- 输入验证、输出编码和参数化查询
- CSRF、XSS、SQL 注入、SSRF 和越权检查
- 文件类型、大小、存储路径和病毒风险控制
- 依赖锁定、漏洞扫描和更新策略
- 管理员操作审计
- 用户数据导出、删除和保留政策
- 备份加密及恢复测试

## 4. 支付与订阅

- 不处理原始银行卡数据，使用合规支付服务
- Webhook 验证签名
- 所有回调可重放且幂等
- 处理失败、取消、退款、争议和过期
- 订单与支付状态不能只依赖前端跳转
- 金额、币种和商品信息由服务端确认

## 5. AI 与工具安全

- 不让 Agent 默认获得生产写权限
- 执行仓库脚本前阅读内容和影响
- 外部文本、Issue、README 和网页可能包含提示注入
- 高风险命令使用沙箱、Allowlist 和人工确认
- 工具输出视为不可信输入并进行验证
- 不在公开 Chat 或仓库中暴露客户数据

## 6. 上线安全门禁

- [ ] 权限矩阵与实际 API 一致
- [ ] 多用户和多租户隔离测试通过
- [ ] Secret 扫描通过
- [ ] 依赖与静态安全扫描通过
- [ ] 关键流程完成独立安全审查
- [ ] 支付和 Webhook 失败路径通过
- [ ] 日志没有敏感数据
- [ ] 备份和恢复真实测试过
