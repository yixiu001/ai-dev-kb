# 编码规则

## 开始编码前

- 明确本次任务和完成条件。
- 找到需要修改的文件与依赖。
- 确认项目技术栈、版本和运行命令。
- 检查是否已有相同功能，避免重复实现。

## 编码时

- 遵循现有项目结构和代码风格。
- 只修改当前任务必要的内容。
- 对外部输入进行校验。
- 对失败情况提供明确错误处理。
- 不把配置、路径和密钥写死在代码中。
- 新增依赖前说明用途、成本和替代方案。
- 不为了“看起来高级”引入不必要抽象。

## 编码后

- 说明修改了哪些文件及原因。
- 提供运行与测试命令。
- 覆盖正常、异常和关键边界场景。
- 检查是否破坏已有功能。
- 无法真实执行时明确写出未验证项。
- 同步更新相关文档、任务和决策。

## 输出代码的最低要求

- 可复制或可直接写入文件。
- 文件路径明确。
- 不省略关键实现。
- 不使用“其余代码不变”隐藏必要上下文。
- 对需要用户替换的值提供清晰占位符。
