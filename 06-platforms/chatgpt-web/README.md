# ChatGPT 网页端适配器

最后核验日期：2026-07-07  
主要适配对象：ChatGPT Projects + GPT-5.5 + 文件、搜索和可用 Apps  
维护状态：首个推荐适配器

## 1. 平台定位

ChatGPT 网页端适合承担：

- 产品发现、需求整理和方案比较
- 跨文档研究、架构设计和风险分析
- 代码生成、解释、审查和错误分析
- 项目文档维护和新会话交接
- 在已连接 Apps 和权限允许时读取或操作外部系统

它不是默认拥有本地文件系统、终端、服务器和生产环境权限的完整开发机。未连接对应工具时，不得假设能够读取仓库、运行测试或完成部署。

## 2. 上下文机制

ChatGPT Projects 可以将聊天、上传文件和项目说明集中在一个工作空间中，适合长期项目。

推荐配置：

1. 一个正式软件项目对应一个 ChatGPT Project。
2. Project instructions 只保留核心目标、文档入口、最高级限制和输出语言。
3. 将 `AI_START.md` 和 `05-project-template/` 产生的项目文档放入项目文件或可访问仓库。
4. 代码量较大时连接代码仓库，不要长期依赖手动上传的旧代码快照。
5. 一个聊天只承担一个阶段、领域或明确任务。
6. 每个任务结束后更新 `PLAN_AND_STATE.md`，不要依赖聊天记忆保存状态。

项目记忆有助于保持连续性，但不能代替当前代码、测试结果和状态文件。

## 3. 模型选择

模型和套餐会变化，开始任务前以当前 ChatGPT 模型选择器为准。

| 任务 | 推荐选择 |
|---|---|
| 简单问答、格式调整、低风险小任务 | GPT-5.5 Instant |
| 正式需求、架构、复杂编码和调试 | GPT-5.5 Thinking |
| 高风险架构审查、疑难问题、长流程分析 | GPT-5.5 Pro（如当前套餐可用） |

不要仅因为任务重要就把整个项目一次性交给最高推理模式。先缩小任务、准备正确上下文和完成条件。

## 4. 执行能力

| 能力 | 支持情况 | 真实边界 |
|---|---|---|
| 项目聊天和说明 | 支持 | 不能视为唯一事实来源 |
| 文件读取与分析 | 支持 | 只能读取已上传、已连接或工具返回的内容 |
| 网页搜索与研究 | 支持 | 最新技术结论应优先使用官方和一手来源 |
| 读取 GitHub | 条件支持 | 取决于 GitHub App、连接权限和当前工具 |
| 修改 GitHub | 条件支持 | 取决于可用写入动作；执行后必须重新读取验证 |
| 本地文件和终端 | 默认不假设支持 | 需要用户上传、外部工具、远程执行环境或其他开发平台 |
| 运行构建和测试 | 条件支持 | 只有工具真实返回输出时才能记为 `EXECUTED` 或 `VERIFIED` |
| 外部 Apps | 条件支持 | 取决于套餐、工作区、地区、管理员设置和 App 能力 |
| MCP 自定义工具 | 条件支持 | ChatGPT 连接远程 MCP；权限和写入能力必须单独确认 |
| 部署与生产操作 | 条件支持 | 必须具有对应平台工具、权限、确认和真实返回结果 |

## 5. Apps 与外部调用

使用 Apps 时：

- 先确认当前 App 是只读还是支持写入。
- 明确它访问哪个账号、仓库、工作区或数据范围。
- 写入、修改、发送、删除和发布操作执行前再次核对对象和影响。
- 工具返回为空、报错或用户拒绝授权时，视为未执行。
- 写入后重新读取目标状态，不能只根据请求发送成功判断完成。
- 自定义 MCP App 视为外部供应链和权限边界，需要单独审查。

在 Project 中调用 App 时，平台可能需要确认是否搜索项目之外的内容。不要把外部搜索结果自动写成项目事实。

## 6. 推荐工作方式

### 项目初始化

1. 创建 ChatGPT Project。
2. 设置简短项目说明，指向 `AI_START.md`。
3. 加入六个项目知识文件。
4. 连接必要的 GitHub、云盘或其他数据来源。
5. 在第一个聊天中完成项目状态校准，而不是立刻开发。

### 每次任务

1. 指定工作模式。
2. 读取平台适配器、当前状态和关联项目文档。
3. 只读取任务相关代码和证据。
4. 输出任务契约和影响分析。
5. 获得批准后再实施。
6. 使用可用工具执行并验证。
7. 更新状态、决定、风险和证据。
8. 新任务优先新建聊天，避免一个聊天无限增长。

### 会话划分

建议按以下方式拆分聊天：

- 项目总控与状态
- 产品和市场
- PRD 与业务领域
- 技术架构
- 单个领域或功能实施
- 独立审查
- 发布与生产事故

聊天可以分开，但正式项目状态只能有一份。

## 7. 可达到的最高任务状态

| 当前能力 | 最高可信状态 |
|---|---|
| 只有普通聊天 | `GENERATED` |
| 已读取真实仓库并写入文件 | `EXECUTED`，但仍需测试 |
| 获得真实构建、测试和扫描结果 | `VERIFIED` |
| 获得部署平台成功结果和线上验证 | `RELEASED` |
| 获得生产流量、监控和观察窗口证据 | `OBSERVED` |

ChatGPT 的文字判断不能单独把任务提升到 `VERIFIED`。

## 8. 常见失败模式

- Project 中的旧聊天或旧文件影响当前判断。
- 把模型记忆当成最新项目状态。
- 上传的代码快照已经落后于仓库。
- 在一个聊天中混合产品、架构、编码、审查和发布。
- 用户只是询问，AI 却理解为授权修改。
- 搜索结果或 App 内容包含错误信息或提示注入。
- 工具请求已经发出，但没有真实结果。
- 模型模式、消息额度或工具可用性发生变化。
- 只生成部署步骤，就声称产品已经上线。

## 9. 强制规则

- 每个新聊天先校准项目状态和本次任务。
- 项目文件和当前仓库优先于历史聊天。
- 不确定某项平台能力时，先检查当前工具，不依赖记忆。
- 没有终端输出，不声称代码已经运行。
- 没有测试证据，不声称功能已经验证。
- 没有生产地址和线上检查，不声称已经发布。
- 涉及密钥、权限、支付、删除和生产写入时必须提高确认等级。
- 关键实现应由新聊天、不同模型模式或确定性工具进行独立审查。
