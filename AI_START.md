# AI 项目工作入口

本文件适用于使用任意 AI 开发平台协助复杂项目开发的所有会话或 Agent 任务。

平台只负责执行方式，通用核心负责项目标准。开始工作前，必须先确定当前 AI 平台，并读取 `06-platforms/` 中对应的适配器。

## 一、先确定平台和工作模式

开始前明确：

- 当前 AI 平台：ChatGPT / Claude Code / Cursor / Gemini CLI / GitHub Copilot / 其他
- 当前可用能力：聊天、文件、代码仓库、终端、网页搜索、外部工具、写入权限
- 当前平台适配器：`06-platforms/<platform>/README.md`

然后确定工作模式：

| 模式 | 允许行为 |
|---|---|
| `DISCOVER` | 研究市场、用户、问题和限制，不写代码 |
| `DEFINE` | 整理需求、业务规则和验收标准，不写代码 |
| `DESIGN` | 设计领域、架构、数据、接口和风险，不实施 |
| `IMPLEMENT` | 在批准范围内修改代码或生成明确补丁 |
| `VERIFY` | 运行或分析测试、日志、扫描和验收结果 |
| `RELEASE` | 执行部署准备、迁移、发布和线上验证 |
| `OPERATE` | 处理监控、事故、用户反馈和迭代 |
| `ANSWER` | 只回答问题，不修改任何文件或状态 |

用户提问不等于授权修改。只有明确进入 `IMPLEMENT` 或 `RELEASE`，且平台具备相应权限时，才允许产生写入操作。

## 二、读取顺序

1. 当前平台的 `06-platforms/<platform>/README.md`
2. 当前项目的 `PROJECT.md`
3. `PRD.md`
4. `ARCHITECTURE.md`
5. `DECISIONS_RISKS_EVIDENCE.md`
6. `PLAN_AND_STATE.md`
7. 当前任务关联的代码、测试、Issue、PR 和日志
8. `RELEASE.md`，仅在部署和生产任务中读取

不要一次性把整个仓库塞入上下文。先根据任务定位领域、模块、接口和文件，再读取相关内容。

## 三、规则优先级

1. 真实生产状态、测试结果和工具返回
2. 当前代码和配置
3. 已批准的正式决定
4. PRD 与架构文档
5. 当前任务契约
6. 通用 AI 协作规则
7. 平台适配规则
8. 历史聊天和 AI 推断

平台适配器只能补充执行细节，不得降低通用核心中的安全、质量、验证和证据要求。

## 四、开始任务前必须输出

- 当前 AI 平台及适配器
- 当前模式
- 平台可用能力与不可用能力
- 任务目标
- 已确认范围
- 需要读取的文件或外部信息
- 可能受影响的模块、接口、数据和测试
- 明确不修改的内容
- 完成条件和验证方法
- 当前假设与待确认事项

## 五、任务状态必须严格区分

```text
PROPOSED  已提出，未批准
APPROVED  已批准，未实施
GENERATED 已生成代码或文档，未执行
EXECUTED  已运行，但结果未完整验证
VERIFIED  验收标准和检查均通过
RELEASED  已部署到目标环境
OBSERVED  已经过线上真实流量或生产观察
```

没有证据时不得提升状态。平台没有真实执行能力时，最高只能标记到 `GENERATED`，除非获得外部工具返回的真实执行证据。

## 六、硬性停止条件

遇到以下情况必须停止实施并报告：

- 需求与已批准决定冲突
- 需要修改未授权模块
- 平台不具备所需读取、执行或写入能力
- 需要删除数据或执行不可逆迁移
- 涉及生产密钥、支付、权限或用户隐私但缺少设计
- 测试失败原因不明确
- 发现现有架构与文档严重不一致
- 工具输出异常、权限不明或外部状态无法确认
- 任务过大，无法形成可审查的小变更

## 七、完成任务后的最低输出

1. 实际修改内容
2. 未修改内容
3. 使用的平台能力、命令和工具
4. 测试与验证结果
5. 证据位置
6. 剩余风险
7. 需要更新的项目文档
8. 下一步唯一建议

## 八、更换 AI 平台

更换平台时不得重建项目事实。新平台应：

1. 读取自己的适配器。
2. 按本文件读取项目知识包。
3. 复述当前阶段、任务、限制和证据缺口。
4. 说明它与上一个平台在能力和权限上的差异。
5. 用户确认理解正确后再继续。

研究依据：[R1]、[R3]、[C3]、[C10]，见 [`00-research/REFERENCES.md`](./00-research/REFERENCES.md)。
