# 领域模型与需求追踪

## 1. 为什么需要领域模型

大型系统中的 AI 错误常来自只理解页面和文件，不理解业务对象、关系和不变量。

每个项目必须定义：

- 核心实体和值对象
- 实体所属用户、组织或租户
- 生命周期和状态转换
- 不允许被破坏的业务不变量
- 谁可以执行什么操作
- 哪些操作需要审计

## 2. 需求追踪链

```text
目标 OBJ
→ 指标 KPI
→ 能力 CAP
→ 需求 RQ
→ 业务规则 BR
→ 领域对象 DM
→ 接口 API
→ 任务 TASK
→ 测试 TEST
→ 发布 REL
```

每个关键功能都必须能够沿链路向前解释价值，向后找到实现和验证。

## 3. 最小追踪表

| 需求 ID | 价值/指标 | 业务规则 | 领域与模块 | 接口/数据 | 任务 | 测试 | 发布 |
|---|---|---|---|---|---|---|---|
| RQ-001 | KPI-001 | BR-001 | Billing | API-003 | TASK-010 | TEST-021 | REL-002 |

## 4. 影响分析用法

任何变更先查追踪表：

- 修改业务规则时，找到相关需求、状态、接口和测试。
- 修改数据结构时，找到迁移、API、任务和回滚。
- 删除功能时，确认没有其他能力和指标依赖。
- 修复缺陷时，新增能复现缺陷的测试并关联原需求。

## 5. 领域边界

按业务责任划分模块，而不是按页面或 AI 生成批次划分。示例：

- Identity：身份与会话
- Access：角色、权限、租户隔离
- Billing：计划、订阅、支付、退款
- Core Domain：产品核心价值
- Notification：邮件、短信、站内消息
- Admin：审核、客服、运营和审计

模块只能通过明确接口协作，不能随意读写其他模块内部数据。
