# AI 开发大型项目调研结论

调研时间：2026-07-07

本调研覆盖 ChatGPT、Claude Code、Cursor、GitHub Copilot、Gemini CLI、Replit、Lovable、多 Agent 工作流，以及软件工程研究。论坛内容属于真实用户经验，不等同于严格统计；当同一问题在多个社区、工具和研究中重复出现时，将其视为高可信工程模式。

## 结论总览

| 问题 | 社区与研究观察 | V1 解决机制 |
|---|---|---|
| 全仓库上下文幻觉 | 大型仓库中，模型难以自动找到真正相关文件；上下文越大不等于理解越好 | 分层上下文、任务上下文包、按领域检索 |
| 静态规则过多 | 规则文件会被遵守，但冗余要求可能增加成本并降低成功率 | 根规则只保留不变量；细节按任务加载 |
| 上下文过期 | 长会话和多聊天容易忘记决定或被旧主题锚定 | `PLAN_AND_STATE.md`、时间戳、任务结束强制交接 |
| 意图误判和越权修改 | 用户提问时，Agent 可能直接编辑、重构或改变执行路径 | 显式工作模式、写权限门禁、修改范围契约 |
| 架构漂移 | Agent 能解决局部问题，但会绕过现有边界、复制逻辑或改变不变量 | 领域边界、架构决定、依赖规则、影响分析 |
| 大改动失败率高 | 未合并的 Agent PR 往往修改更多文件、CI 失败更多 | 小型垂直切片、变更预算、逐步合并 |
| 错误声称完成 | 研究发现约束违反和不准确进度报告仍是突出问题 | 七级状态模型、证据账本、独立验证 |
| 测试不足或测试驱动错误修改 | Agent 可能修改正确生产代码去迁就错误测试，也可能只验证快乐路径 | 测试权威层级、质量门禁、需求到测试追踪 |
| 原型与生产脱节 | 社区常见开发环境正常、发布后 API、密钥、端口和数据失效 | 环境矩阵、可重复构建、发布 Runbook、冒烟测试 |
| 安全和租户隔离缺失 | Vibe-coded 产品常见授权、RLS、密钥、支付 Webhook 和日志问题 | 威胁建模、权限矩阵、租户隔离测试、安全门禁 |
| 多 Agent 冲突 | Worktree 可隔离工作目录，但无法解决同一模块的语义冲突 | 模块所有权、独立任务、Worktree、单一集成人 |
| 工具和集成不可靠 | AI 编程工具大量问题来自 API、配置、终端和命令执行 | 沙箱、超时、重试、权限矩阵、工具结果复核 |
| 非技术用户审查负担高 | 新手借助 AI 产生更大变更，接受率更低、审查意见更多 | 阶段门禁、限制变更规模、关键领域强制专家审查 |
| 成本与上下文浪费 | 长上下文、多工具和重复探索可显著增加成本 | 任务预算、上下文预算、复用已验证证据 |

## 关键判断

### 1. 大模型不应被要求“记住整个项目”

复杂项目应把稳定事实、动态状态和本次任务分开。AI 每次只读取当前任务需要的领域、接口、数据和代码。大型上下文适合检索，不适合无差别注入。

### 2. 一个巨大的 AGENTS.md 不是答案

研究显示，自动生成或冗长的仓库说明不一定提高任务成功率，却会增加推理步骤和成本。根级 AI 规则应只包含安全不变量、运行命令、禁止事项和文档入口；业务与模块细节放在对应文档中按需读取。

### 3. AI 必须受到变更控制，而不只是提示词约束

“请不要乱改”不足以保护大型项目。每个任务必须有允许修改范围、禁止范围、文件或模块所有权、验收标准、回滚方式和证据要求。

### 4. 生产级价值来自最后 20%

论坛中的重复案例表明，AI 很容易做出可演示的界面和快乐路径；真正产品化需要授权、失败处理、数据一致性、支付边界、监控、迁移、备份和客服支持。

### 5. 多 Agent 是协调问题，不只是并行问题

默认应使用单 Agent 顺序开发。只有任务能够按领域或模块真正独立时才并行。每个 Agent 使用独立分支和 Worktree，并由一个集成人按顺序合并和运行完整门禁。

### 6. “通过测试”仍不是最终真相

测试可能缺失、错误或只覆盖局部。验证必须同时对照需求、业务不变量、自动测试、集成环境和真实生产观察。

## V1 设计决定

本知识库因此不再以“功能说明”为中心，而采用以下五个支柱：

1. AI 协作与项目状态
2. 产品和需求定义
3. 生产级系统设计
4. 小步开发与质量门禁
5. 发布、运维与市场反馈
