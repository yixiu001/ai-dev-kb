# Git 与 GitHub 常见错误和恢复

> 面向：遇到推送失败、冲突、误删或历史混乱的用户

## 1. `nothing to commit`

说明当前没有新的已跟踪修改，或文件被 `.gitignore` 排除。

检查：

```bash
git status
git check-ignore -v <文件>
```

## 2. `non-fast-forward`

远程分支包含本地没有的 Commit。

安全处理：

```bash
git fetch origin
git log --oneline --graph --all
git pull --ff-only
```

如果本地也有新 Commit，需要选择 merge 或 rebase，不要直接强制推送覆盖远程历史。

## 3. 合并冲突

先查看：

```bash
git status
```

打开冲突文件，理解双方修改目的，完成编辑后：

```bash
git add <文件>
git commit
```

如果正在 rebase：

```bash
git add <文件>
git rebase --continue
```

放弃当前合并或 rebase：

```bash
git merge --abort
git rebase --abort
```

## 4. 提交到了错误分支

尚未 Push 时，可以先创建正确分支保留 Commit：

```bash
git branch feature/correct-branch
git switch main
git reset --hard origin/main
```

`reset --hard` 会删除当前分支未保存修改，执行前必须确认状态并建立备份分支。

## 5. Commit 消息写错

最近 Commit 尚未共享：

```bash
git commit --amend -m "正确消息"
```

已经 Push 到共享分支时，通常保留历史或通过新的说明处理，不要随意改写历史。

## 6. 忘记添加文件

最近 Commit 尚未共享：

```bash
git add <遗漏文件>
git commit --amend --no-edit
```

## 7. 误删本地文件

文件尚未 Commit：

```bash
git restore <文件>
```

如果删除已经 Commit，可以从历史恢复：

```bash
git restore --source=<commit> <文件>
```

## 8. 误删分支或误 reset

查看操作历史：

```bash
git reflog
```

找到目标 Commit 后：

```bash
git branch recovery/<名称> <commit>
```

先创建恢复分支，再继续其他操作。

## 9. 想撤销已经进入共享历史的 Commit

使用：

```bash
git revert <commit>
```

它会创建一个反向 Commit，不会重写其他人的历史。

## 10. 推送被仓库规则阻止

可能原因：

- main 禁止直接 Push；
- Required checks 未通过；
- 缺少 Review；
- Commit 未签名；
- 分支过期；
- Secret scanning 阻止推送。

不要绕过规则。创建 PR、修复检查或处理安全问题。

## 11. GitHub 检测到 Secret

立即：

1. 停止继续推送；
2. 撤销或轮换真实密钥；
3. 从当前文件删除；
4. 判断是否需要清理历史；
5. 检查访问日志和滥用；
6. 增加 `.gitignore`、Secret 和扫描规则。

密钥从代码中删除，不代表已经安全；已暴露的密钥必须失效。

## 12. 大文件 Push 失败

检查文件是否应：

- 从 Git 历史移除；
- 使用 Git LFS；
- 放到 Releases；
- 放到对象存储；
- 通过下载脚本获取。

不要把 `node_modules`、数据库、视频素材和用户上传目录直接加入 Git。

## 13. `detached HEAD`

说明当前停留在某个 Commit，而不是正常分支。

需要保留修改时：

```bash
git switch -c recovery/my-work
```

不要在 detached HEAD 中长期开发后直接离开。

## 14. 本地和 GitHub 文件不一致

检查：

```bash
git status
git branch -vv
git remote -v
git fetch origin
git log --oneline --graph --all
```

常见原因：当前分支不同、未 Push、未 Pull、远程地址错误或查看了错误仓库。

## 15. Actions 失败

查看：

- 失败 Job 和 Step；
- 完整日志；
- 运行使用的 Commit；
- 环境变量和 Secret 名称；
- 运行时版本；
- 缓存和依赖锁文件；
- 权限设置。

先在本地复现相同命令，不要反复空跑工作流消耗额度。

## 通用恢复原则

```text
停止继续修改
→ 保存错误和状态
→ 找到最后可用Commit
→ 建立恢复分支
→ 最小化复现
→ 一次验证一个根因
→ 最小修复
→ 回归测试
```
