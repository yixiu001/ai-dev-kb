# Git 与 GitHub 有什么区别

> 面向：第一次接触代码版本管理的用户

## 一句话理解

- **Git**：安装在我的电脑上，用来记录代码每一次变化。
- **GitHub**：互联网上的代码托管和协作平台，用来保存远程仓库、审查代码、管理任务和自动化流程。

它们不是同一个东西，也不能互相完全替代。

## 一个生活化比喻

我可以把项目理解成一本正在不断修改的书：

- Git 是我电脑里的“版本历史系统”；
- Commit 是一次带说明的存档；
- Branch 是一本书的独立修改副本；
- GitHub 是云端书库；
- Pull Request 是我提交的一份“请审查这次修改”的申请；
- Merge 是把审查通过的修改合并进正式版本。

## Git 在本地做什么

即使断网，Git 仍然可以：

- 创建仓库；
- 查看文件变化；
- 保存 Commit；
- 创建和切换分支；
- 比较版本；
- 合并修改；
- 撤销错误；
- 恢复历史版本。

## GitHub 在云端做什么

GitHub 主要提供：

- 公共和私有远程仓库；
- Pull Request 和代码审查；
- Issues、Projects、Milestones；
- Actions 自动构建、测试和部署；
- Pages 静态网站；
- Packages 软件包存储；
- Codespaces 云端开发环境；
- Dependabot、安全扫描和密钥保护；
- 团队、权限、规则和审计；
- Releases 和 Tags 发布管理。

## 本地仓库和远程仓库

```mermaid
flowchart LR
    W[工作目录<br/>我正在编辑的文件] --> S[暂存区<br/>准备提交的修改]
    S --> L[本地Git仓库<br/>Commit历史]
    L -- push --> R[GitHub远程仓库]
    R -- pull/fetch --> L
```

### 工作目录

我在 VS Code 中实际看到和修改的文件。

### 暂存区

我通过 `git add` 选择准备放入下一次 Commit 的修改。

### 本地仓库

我通过 `git commit` 保存的版本历史。

### 远程仓库

我通过 `git push` 上传到 GitHub 的版本。

## GitHub Desktop、Git CLI 和网页端有什么区别

| 方式 | 适合什么 | 优点 | 限制 |
|---|---|---|---|
| Git 命令行 | 长期开发、复杂操作 | 功能完整、通用、适合 AI 和自动化 | 新手需要学习命令 |
| GitHub Desktop | 日常提交、分支和同步 | 界面直观 | 高级操作仍可能需要命令行 |
| GitHub 网页端 | 小文件编辑、Issue、PR、设置 | 无需本地安装 | 不适合大量代码修改和本地测试 |
| IDE 内置 Git | 边写代码边提交 | 与代码编辑结合紧密 | 不同 IDE 界面不同 |

我可以同时使用它们。它们操作的是同一套 Git 历史。

## GitHub 不是备份系统的全部

GitHub 能保存已经提交并推送的代码，但不能自动保护：

- 尚未 Commit 的本地文件；
- 被 `.gitignore` 排除的数据；
- 生产数据库；
- 用户上传文件；
- 外部服务配置；
- 没有推送的本地分支。

正式项目仍然需要数据库备份、对象存储备份和配置管理。

## 我什么时候必须使用 Git

只要项目会持续修改，我就应该尽早使用 Git，特别是：

- AI 会修改多个文件；
- 我需要尝试不同方案；
- 项目需要上线；
- 多个人或多个 AI 同时工作；
- 我需要知道是谁在什么时候改了什么；
- 修改失败后需要恢复。

## 我怎样判断自己已经理解

我应该能回答：

1. `git commit` 保存到哪里？——本地 Git 历史。
2. `git push` 做什么？——把本地 Commit 上传到远程仓库。
3. GitHub 上有代码，是否代表本地最新？——不一定，需要同步。
4. 本地文件改了但没 Commit，GitHub 能看到吗？——不能。
5. AI 说“已经提交”，我需要什么证据？——Commit SHA、分支或 PR。
