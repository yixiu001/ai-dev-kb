# 权限、安全与仓库保护

> 面向：需要保护商业代码、用户数据和生产流程的用户

## 最小权限原则

每个人和每个 AI 工具只获得完成当前任务需要的权限。

常见权限层级：

- Read：查看和克隆；
- Triage：管理 Issue 和 PR，不修改代码；
- Write：推送代码；
- Maintain：管理仓库但不能执行部分高风险管理操作；
- Admin：完整仓库管理权限。

不要为了方便把外包、临时成员或 AI 工具长期设为 Admin。

## 个人仓库与 Organization

商业项目开始多人协作后，建议使用 Organization：

- 成员离开时可以集中移除权限；
- 可以使用 Teams 分组；
- 仓库不依赖某个个人账号；
- 计费、策略和审计更集中；
- 可以分别管理开发、审查和发布权限。

## 保护 main

推荐至少设置：

- 必须通过 Pull Request；
- 必须通过自动检查；
- 必须解决 Review 讨论；
- 禁止删除 main；
- 限制谁能直接修改；
- 必要时要求 Code Owner 审查。

如果套餐不支持某项私有仓库高级规则，就通过流程、PR 和 Actions 尽量补足，但不能假装已经强制执行。

## CODEOWNERS

`CODEOWNERS` 可以为特定目录指定负责人：

```text
/docs/ @docs-team
/src/billing/ @billing-team
/.github/workflows/ @platform-team
```

当相关文件发生修改时，可以自动请求对应人员审查。私有仓库中的强制审批能力取决于套餐。

## 密钥保护

绝不能提交：

- API Key；
- Access Token；
- Cookie；
- 私钥；
- 数据库密码；
- 生产 `.env`；
- 云平台凭证。

正确做法：

- 本地使用环境变量；
- 仓库只提交 `.env.example`；
- Actions 使用 GitHub Secrets；
- 生产使用部署平台 Secret；
- 定期轮换高权限凭证。

如果密钥已经进入 Commit，删除当前文件还不够。必须立即撤销或轮换密钥，并处理 Git 历史。

## Dependabot

Dependabot 可以：

- 提醒依赖漏洞；
- 自动创建安全更新 PR；
- 创建版本更新 PR。

自动更新 PR 仍需运行测试和审查，不应直接自动合并所有大版本升级。

## Code scanning 与 Secret protection

GitHub 提供不同级别的：

- Code scanning；
- Secret scanning；
- Push protection；
- Dependency review；
- Security overview。

公共仓库、私有仓库、组织套餐和 Advanced Security 附加产品的可用范围不同，应核对当前计划。

## Actions 安全

- 第三方 Action 使用固定版本或 Commit；
- 限制 `GITHUB_TOKEN` 权限；
- 不把 Secret 打印到日志；
- Fork PR 不应自动获得生产 Secret；
- 生产发布使用 Environment 和审批；
- Self-hosted runner 不运行不可信代码；
- 定期更新 Actions 依赖。

## 审计和记录

正式项目应能追踪：

- 谁修改了代码；
- 谁批准 PR；
- 哪个工作流发布；
- 哪个版本进入生产；
- 谁修改权限和 Secret；
- 事故发生时的操作时间线。

Enterprise 提供更强的集中审计和身份能力，小团队也应保留 PR、Commit、Actions 和发布记录。

## 仓库安全检查

- [ ] 已启用两步验证；
- [ ] main 有保护规则；
- [ ] 成员权限符合职责；
- [ ] 不存在长期不用的 Token；
- [ ] `.gitignore` 排除敏感文件；
- [ ] Dependabot alerts 已开启；
- [ ] Actions 使用最小权限；
- [ ] 生产部署有独立环境；
- [ ] 高风险目录有审查负责人；
- [ ] 密钥泄露有撤销和处理流程。
