# 实施流程与质量门禁

## 1. 使用垂直切片

一个切片应从用户行为贯穿到数据和验证，例如：

```text
页面/客户端 → API → 领域规则 → 数据 → 日志 → 测试
```

不要先生成整个前端，再生成整个后端，最后才发现接口和业务规则不一致。

## 2. 标准实施循环

1. 选择一个已批准任务。
2. 读取关联需求、领域、架构、代码和测试。
3. 输出影响分析和实施计划。
4. 用户或项目规则批准。
5. 建立分支或 Worktree。
6. 先建立复现测试或验收脚本。
7. 实施最小变更。
8. 运行局部测试。
9. 运行项目质量门禁。
10. 独立审查 Diff。
11. 合并并更新项目状态和证据。

## 3. 质量门禁顺序

```text
格式与静态检查
→ 构建与类型检查
→ 单元测试
→ 集成测试
→ 关键端到端测试
→ 安全和依赖扫描
→ 性能或迁移检查
→ 人工/独立 AI 审查
```

所有门禁尽量使用确定性工具，而不是只让同一个模型评价自己的代码。

## 4. 测试权威层级

发生冲突时按以下顺序分析，而不是盲目改代码：

1. 已批准业务不变量和验收标准
2. 可复现的真实缺陷
3. 集成与端到端行为
4. 单元测试
5. AI 推断

测试本身可能错误。修改正确生产逻辑去适配错误 Mock 或过时断言前，必须解释冲突并获得批准。

## 5. 审查重点

- 是否满足关联需求
- 是否出现无关修改
- 是否复用现有逻辑而非复制
- 是否破坏接口和数据兼容
- 是否遗漏权限、失败和并发路径
- 是否存在临时代码或未处理 TODO
- 是否更新测试、日志和文档

## 6. 完成定义

任务只能标记 `VERIFIED`，当：

- 验收标准逐条有证据
- 所有必要门禁通过
- Diff 已审查
- 状态、决定和风险已更新
- 没有隐藏的“稍后处理”阻塞项
